# filefunc — Manual for AI Agents

This document is the reference manual for AI code agents (Claude Code, etc.) when writing code in filefunc projects.

---

## How to Navigate

### 1. Read codebook.yaml first

The `codebook.yaml` at the project root is the project's vocabulary. Check feature and type values, then compose grep queries.

### 2. Find target files with grep

```bash
rg '//ff:func feature=validate'            # all funcs in validate feature
rg '//ff:func feature=validate type=rule'  # validate rules only
rg '//ff:type feature=parse'               # all types in parse feature
```

### 3. Read top annotations to narrow down

Each file's `//ff:what` tells you what it does without reading the body.

### 4. Full read only the files you need, then work

## Code Writing Rules

### File Creation

- **func file**: filename = func name (snake_case). `check_one_file_one_func.go` → `CheckOneFileOneFunc`
- **type file**: filename = type name (snake_case). `violation.go` → `Violation`
- **method file**: `{receiver}_{method}.go`. `server_start.go` → `(Server).Start`
- **One func/type/method per file.** Helper functions go in separate files.
- init() is only allowed alongside a var or func. Standalone init() files are forbidden.

### Annotations

Write `//ff:func` or `//ff:type` and `//ff:what` at the **very top** of every func/type file (above the package declaration). These are manual.

`//ff:calls`, `//ff:uses`, `//ff:checked` are auto-generated by tools. Do not write them manually.

```go
//ff:func feature=validate type=rule
//ff:what F1: validates one func per file
package validate
```

```go
//ff:type feature=validate type=model
//ff:what holds validation violation results
package model
```

| Annotation | Required | Generated by | Description |
|---|---|---|---|
| `//ff:func` | Yes (func files) | Manual | Metadata: feature, type, etc. Values must exist in codebook.yaml |
| `//ff:type` | Yes (type files) | Manual | Metadata: feature, type, etc. Values must exist in codebook.yaml |
| `//ff:what` | Yes (func/type files) | Manual | One-line description. What does this do? |
| `//ff:why` | No | Manual | Why was it designed this way? Only record user decisions/requirements |
| `//ff:calls` | Auto | `filefunc annotate` | Project-internal function calls (from Go AST) |
| `//ff:uses` | Auto | `filefunc annotate` | Project-internal type usages (from Go AST) |
| `//ff:checked` | Auto | `filefunc llmc` | LLM verification signature (llm=model hash=bodyhash) |

### Code Quality

- **Nesting depth ≤ 2.** One loop + one branch is the maximum. Beyond that, extract a function or merge conditions.
- **Func recommended ≤ 100 lines, hard limit 1000 lines.**
- Early return pattern. Handle errors immediately and return.
- gofmt compliance.

### Naming

- Filenames: `snake_case` (`check_one_file_one_func.go`)
- Variables/functions: `camelCase` (`maxDepth`, `walkGoFiles`)
- Types: `PascalCase` (`GoFile`, `Violation`)

## Codebook Values (Current)

Codebook has two sections: `required` and `optional`.

**Required keys** must be present in every `//ff:func` and `//ff:type` annotation. This guarantees grep reliability.

**Optional keys** are used only when relevant.

```
required:
  feature: validate, annotate, chain, parse, codebook, report, cli
  type:    command, rule, parser, walker, model, formatter, loader, util

optional:
  pattern: error-collection, file-visitor, rule-registry
  level:   error, warning, info
```

If you need a new value, amend codebook.yaml.

### Codebook format rules

- `required` section must have at least one key with at least one value
- No duplicate values within the same key
- All values must be lowercase + hyphens only (`[a-z][a-z0-9-]*`)
- Empty arrays are not allowed — remove the key instead
- Codebook is validated first. If codebook fails, code validation does not run.
- All required keys must be present in every annotation (A8). Missing → ERROR.

## Workflow

### After writing code

`codebook.yaml` must exist in the project root (next to `go.mod`). Validate auto-detects it. Without it, validate will error.

```bash
# 1. Validate structure rules (codebook.yaml auto-detected)
filefunc validate ./internal/

# 2. Auto-generate calls/uses
filefunc annotate ./internal/

# 3. Verify what-body match with LLM (optional, requires ollama)
filefunc llmc ./internal/
```

### Validate options

```bash
filefunc validate --format json ./internal/                          # JSON output
filefunc validate --codebook /path/to/codebook.yaml ./internal/      # override codebook path
```

### LLM verification options

```bash
filefunc llmc --model qwen3:8b ./internal/      # different model
filefunc llmc --threshold 0.9 ./internal/        # stricter threshold
```

Exit code 1 on violations. Zero violations required before committing.

## Exceptions (Not Violations)

- `_test.go` files may have multiple funcs (F5)
- Unexported types alongside a func in the same file are allowed (F6)
- Semantically grouped consts in one file are allowed (F7)
- const-only and var-only files do not require annotations
- If no `//ff:checked` exists in the project, A7 (hash check) is skipped entirely

## Common Mistakes and Fixes

| Mistake | Cause | Fix |
|---|---|---|
| Two funcs in one file | Habit | Extract helper functions into separate files |
| for → switch → if (depth 3) | Nested branching | Use type assertions + early continue, or extract inner logic to a func |
| for → if → if (depth 3) | Nested conditions | Merge conditions `if a && b`, or use early break |
| Missing //ff:what | Forgot annotation | Write annotations first when creating a file |
| Value not in codebook | Didn't check vocabulary | Check codebook.yaml first. If absent, amend codebook |
| Manually writing //ff:calls | Habit | Run `filefunc annotate` instead. calls/uses are auto-generated |
| //ff:checked hash mismatch | Body changed after llmc | Run `filefunc llmc` again to re-verify |
| "codebook.yaml required" error | No codebook.yaml in project root | Create codebook.yaml next to go.mod. Every filefunc project requires one |
