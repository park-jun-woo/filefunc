# 제약하라, 그러면 수렴한다: 기계 검증 가능 룰을 통한 뉴로-심볼릭 코드 변환

## 초록

LLM 기반 코드 에이전트는 "코드를 개선하라"는 개방형 지시에서 수렴하지 않는다. 우리는 **수렴적 제약(convergent constraint)**이라는 설계 방법론을 제시한다: 기계적으로 검증 가능하고, 이산적 종료 조건을 가지며, 충족 시 유의미한 구조적 속성을 보장하는 룰을 설계하면, LLM은 `while violation > 0: fix` 루프만으로 임의의 코드베이스를 해당 속성으로 수렴시킬 수 있다.

이 방법론의 인스턴스로 filefunc을 제시한다. 22개의 기계 검증 가능 룰(파일 구조, 네스팅 깊이, 제어구조 어노테이션)을 정의하고, 위반 0건 달성 시 모든 함수가 Bohm-Jacopini 3분류(순차/선택/반복) 중 정확히 하나로 수렴함을 증명한다. 10개 Go 오픈소스 프로젝트(총 85K LOC)에 적용하여, LLM 자동 리팩토링으로 평균 N회 반복 내에 위반 0건에 도달하고, 수렴된 코드베이스에서 AI 에이전트의 태스크 완수율이 X% 향상됨을 보인다.

핵심 통찰: 모델을 더 크게 만드는 것이 아니라, 모델을 더 작은 역할에 가두는 기호 시스템을 설계하는 것이 뉴로-심볼릭 소프트웨어 엔지니어링의 본질이다.

## 1. 문제: LLM은 개방형 지시에서 수렴하지 않는다

### 1.1 "좋은 코드"는 종료 조건이 아니다

AI 코드 에이전트에게 "이 코드를 리팩토링하라"고 지시하면, 에이전트는 무언가를 바꾼다. 그러나:

- 언제 끝인가? 에이전트는 모른다.
- 결과가 충분한가? 판정 기준이 없다.
- 다시 실행하면 같은 결과인가? 아니다.

"코드 품질"은 스펙트럼이다. 스펙트럼에는 종료 조건이 없다. LLM에게 스펙트럼 위의 목표를 주면 수렴이 보장되지 않는다.

### 1.2 기존 도구의 한계

정적 분석 도구(golangci-lint, ESLint, SonarQube)는 룰을 제공하지만:

- 룰 간 우선순위가 모호하다 (warning vs error 혼재)
- 룰 충족이 유의미한 전역 속성을 보장하지 않는다
- "위반 0건"이 목표로 설정되는 경우가 드물다

결과적으로 LLM이 "lint 위반을 고쳐라"라고 지시받아도, 어떤 위반을 무시하고 어떤 위반을 고칠지 판단해야 한다. 판단이 개입하는 순간 수렴이 깨진다.

### 1.3 필요한 것

LLM이 판단 없이 기계적으로 반복 실행하여, 유의미한 목표 상태에 도달하는 시스템. 이를 위해 세 가지 조건이 필요하다:

1. **기계 검증 가능**: 위반 여부를 LLM 없이 판정 (symbolic)
2. **이산적 종료**: 위반 수가 0 또는 양의 정수 (스펙트럼 아님)
3. **속성 보장**: 위반 0건 달성 시 증명 가능한 구조적 속성이 성립

이 세 조건을 만족하는 룰 집합을 **수렴적 제약**이라 정의한다.

## 2. 수렴적 제약 방법론

### 2.1 정의

수렴적 제약 시스템 `C = (R, V, P)`는:

- `R`: 기계 검증 가능한 룰의 유한 집합
- `V: Code → ℕ`: 코드에 대해 위반 수를 반환하는 검증 함수 (LLM 불필요)
- `P`: `V(code) = 0`일 때 성립이 보장되는 구조적 속성의 집합

### 2.2 수렴 루프

```
input: 임의의 코드베이스 S₀
output: V(Sₙ) = 0인 코드베이스 Sₙ

repeat:
    violations = V(Sᵢ)           // symbolic: 기계적 검증
    if violations = 0: return Sᵢ  // 종료
    Sᵢ₊₁ = LLM_fix(Sᵢ, violations) // neural: LLM이 위반 수정
    i = i + 1
```

이 루프에서 LLM의 역할은 **변환 엔진**이다. 무엇이 위반인지는 `V`가 판정하고, 위반이 해소되었는지도 `V`가 판정한다. LLM은 `V`가 지적한 위반을 고치는 일만 한다.

### 2.3 수렴 조건

루프가 종료하려면:

- **단조 감소**: 각 반복에서 `V(Sᵢ₊₁) < V(Sᵢ)` (또는 최소한 새 위반을 도입하지 않음)
- **유한 위반**: 초기 위반 수 `V(S₀)`가 유한

실제로는 LLM이 하나의 위반을 고치면서 새 위반을 도입할 수 있다. 그러나 룰이 **국소적**(한 파일의 위반 수정이 다른 파일에 영향을 주지 않음)이면 단조 감소가 보장된다. filefunc의 룰은 파일 단위로 검증되므로 이 조건을 만족한다.

### 2.4 뉴로-심볼릭 아키텍처

```
┌─────────────────────────────────┐
│         Symbolic System          │
│                                  │
│  ┌──────────┐    ┌───────────┐  │
│  │  Rules R  │    │ Verify V  │  │
│  │ (22 rules)│    │ (AST+grep)│  │
│  └──────────┘    └─────┬─────┘  │
│                        │        │
│         violations     │        │
│         ┌──────────────┘        │
│         ▼                       │
│  ┌─────────────┐                │
│  │   LLM fix   │ ◄── Neural    │
│  │ (변환 엔진)  │    Component  │
│  └─────────────┘                │
│         │                       │
│         ▼ modified code         │
│  ┌───────────┐                  │
│  │ Verify V  │ ◄── 재검증       │
│  └───────────┘                  │
└─────────────────────────────────┘
```

LLM은 시스템 **안에** 있다. 시스템 밖에서 코드를 통째로 이해하는 주체가 아니라, 기호 시스템이 정의한 문제를 풀고 기호 시스템이 결과를 검증받는 구성요소다.

이것이 뉴로-심볼릭의 핵심이다: **기호 시스템이 문제를 정의하고 해를 검증하고, 신경망이 해를 생성한다.**

## 3. 인스턴스: filefunc

### 3.1 룰 집합 R

filefunc은 22개 룰을 정의한다:

**파일 구조 (F1-F4)**: 1파일 1함수/타입/메서드

| 룰 | 내용 | 검증 방법 |
|---|---|---|
| F1 | 파일당 func 1개 | AST func 카운트 |
| F2 | 파일당 type 1개 | AST type 카운트 |
| F3 | 파일당 method 1개 | AST method 카운트 |
| F4 | init() 단독 불허 | AST init + func/var 유무 |

**코드 품질 (Q1-Q3)**: 네스팅 깊이, 함수 길이

| 룰 | 내용 | 검증 방법 |
|---|---|---|
| Q1 | depth <= dimension + 1 | AST depth 측정 |
| Q2 | func <= 1000줄 | 라인 카운트 |
| Q3 | func 권고 100줄 (control별 차등) | 라인 카운트 + control |

**어노테이션 (A1-A16)**: 메타데이터 정합성

| 룰 | 내용 | 검증 방법 |
|---|---|---|
| A1 | func → //ff:func, type → //ff:type 필수 | 패턴 매칭 |
| A2 | 어노테이션 값은 codebook에 존재 | codebook 대조 |
| A9 | func 파일은 control= 필수 | 어노테이션 파싱 |
| A10-A14 | control과 실제 AST 일치 | AST 검증 |
| A15-A16 | iteration은 dimension 필수, 양의 정수 | 값 파싱 |

22개 룰 전부 **LLM 없이** Go AST, 패턴 매칭, codebook 대조만으로 검증된다.

### 3.2 검증 함수 V

```bash
$ filefunc validate ./...
```

출력: `ERROR` 또는 `WARNING` 목록. 위반 수는 정확한 자연수. 0이면 종료.

### 3.3 보장 속성 P

`V(code) = 0`일 때 성립하는 속성:

**P1. 단일 책임**: 모든 파일이 정확히 1개의 func, type, 또는 method를 포함한다.

**P2. Bohm-Jacopini 수렴**: 모든 함수가 sequence, selection, iteration 중 정확히 하나로 분류된다.

증명: A9가 `control=` 필수를 강제하고, A10-A14가 control 값과 AST의 일치를 강제한다. Q1이 depth <= dimension + 1을 강제한다. depth가 제한된 함수에서 깊이 1의 지배적 제어구조는 정확히 하나다 (A12가 sequence에 loop/switch를 금지하고, A13이 selection에 loop을 금지하고, A14가 iteration에 switch를 금지). 따라서 모든 함수는 정확히 하나의 Bohm-Jacopini 범주에 속한다. ∎

**P3. 차원 인식 깊이 제한**: iteration 함수의 네스팅 깊이가 데이터 차원에 비례하여 제한된다.

**P4. 파일명 = 함수명**: F1과 네이밍 룰에 의해, 파일 시스템이 곧 함수 인덱스다.

### 3.4 수렴 루프 실행 예시

```
$ filefunc validate ./...
ERROR F1: multi_func.go has 3 functions (expected 1)
ERROR Q1: deep_func.go depth 4 exceeds limit 2
ERROR A9: no_control.go missing control= annotation
3 errors

→ LLM이 multi_func.go를 3개 파일로 분리
→ LLM이 deep_func.go에서 헬퍼 추출
→ LLM이 no_control.go에 control=sequence 추가

$ filefunc validate ./...
ERROR A12: extracted_helper.go control=sequence but has loop at depth 1

→ LLM이 control=iteration으로 수정

$ filefunc validate ./...
No violations found.
```

2회 반복으로 수렴. LLM은 매 반복에서 "무엇이 잘못인가"를 판단할 필요 없이, validate가 알려주는 ERROR를 고치기만 했다.

## 4. 실험

### 4.1 설정

10개 Go 오픈소스 프로젝트를 포크하여 filefunc 수렴 루프를 실행한다.

| 프로젝트 | 유형 | LOC | 설명 |
|---|---|---|---|
| cobra | CLI 프레임워크 | ~8K | 명령줄 인터페이스 |
| viper | 설정 관리 | ~6K | 설정 파일 파싱 |
| gin | 웹 프레임워크 | ~15K | HTTP 핸들러 |
| goldmark | Markdown 파서 | ~10K | 파서/렌더러 |
| sqlc | 코드 생성기 | ~20K | SQL→Go 코드 생성 |
| chi | HTTP 라우터 | ~3K | 라우팅 |
| lo | 유틸리티 | ~5K | 제네릭 유틸리티 |
| bubbletea | TUI 프레임워크 | ~5K | 터미널 UI |
| air | 핫리로드 | ~3K | 파일 감시+재시작 |
| task | 태스크 러너 | ~10K | 태스크 자동화 |

### 4.2 측정 항목

**수렴 측정**:
- 초기 위반 수 `V(S₀)`
- 수렴까지 반복 횟수
- 반복당 위반 감소율
- 새로 도입된 위반 수 (단조 감소 검증)

**구조 변환 측정** (before → after):
- 파일 수, 평균 LOC/파일
- F1(SRP) 위반률
- Q1(depth) 위반 수
- LOC 증가율

**Bohm-Jacopini 분포**:
- 프로젝트별 sequence/selection/iteration 비율
- dimension 분포 (iteration 함수)
- 분류 불가 함수 수 (0건 예상)

**AI 에이전트 성능** (before/after 각각):
- 동일 수정 태스크 5개의 성공률
- 태스크당 소비 토큰
- 태스크당 읽은 파일 수

**기능 동일성**:
- `go test ./...` 통과 확인

### 4.3 예상 결과

| 지표 | 예상 |
|---|---|
| 수렴 반복 횟수 | 프로젝트당 3-8회 |
| Bohm-Jacopini 분류 불가 | 0건 (P2 보장) |
| LOC 증가율 | 30-60% |
| 파일 수 증가율 | 500-1500% |
| AI 에이전트 태스크 성공률 | before 대비 유의미한 향상 |

### 4.4 프로젝트 유형별 control 분포 가설

| 유형 | sequence | selection | iteration |
|---|---|---|---|
| CLI 프레임워크 (cobra) | 높음 | 낮음 | 중간 |
| 파서 (goldmark) | 낮음 | 중간 | 높음 |
| 웹 프레임워크 (gin) | 높음 | 중간 | 낮음 |
| 코드 생성기 (sqlc) | 높음 | 낮음 | 중간 |
| 유틸리티 (lo) | 낮음 | 낮음 | 높음 |

control 분포가 프로젝트 유형의 지문이라는 가설을 10개 프로젝트로 검증한다.

## 5. 기존 접근과의 비교

### 5.1 LLM 역할의 차이

| 접근 | LLM의 역할 | 수렴 보장 |
|---|---|---|
| "코드를 리팩토링하라" | 판단자 + 실행자 | 없음 |
| "lint 위반을 고쳐라" | 판단자 + 실행자 (어떤 위반?) | 약함 |
| **수렴적 제약** | **실행자만** (판단은 V가) | **보장** |

### 5.2 기존 뉴로-심볼릭과의 차이

| 시스템 | 기호 역할 | 신경 역할 | 도메인 |
|---|---|---|---|
| AlphaGo | 게임 트리 탐색 | 가치/정책 평가 | 바둑 |
| Neuro-symbolic AI (Garcez) | 논리 추론 | 지각/패턴 인식 | 일반 AI |
| Program synthesis | 형식 명세 | 코드 생성 | 프로그래밍 |
| **filefunc** | **룰 검증 + 종료 판정** | **코드 변환** | **소프트웨어 공학** |

filefunc의 차별점: 기호 시스템이 **종료 조건**을 정의하고, 신경망의 출력을 **매 반복 검증**한다. 기존 뉴로-심볼릭이 주로 추론에 초점을 맞추는 반면, 우리는 **변환의 수렴**에 초점을 맞춘다.

### 5.3 형식 검증과의 차이

| 측면 | 형식 검증 | 수렴적 제약 |
|---|---|---|
| 검증 대상 | 프로그램의 정확성 | 코드의 구조적 속성 |
| 명세 언어 | 수학적 논리 | 22개 AST 룰 |
| 검증 비용 | 높음 (정리 증명) | 낮음 (grep + AST) |
| 자동 수정 | 불가 | LLM으로 가능 |
| 보장 수준 | 기능적 정확성 | 구조적 속성 |

수렴적 제약은 형식 검증의 무거움 없이, 구조적 속성에 한정하여 "검증 + 자동 수정" 루프를 달성한다.

## 6. 범용성

### 6.1 코드 외 도메인

수렴적 제약 방법론은 코드에 한정되지 않는다. 세 가지 조건(기계 검증 가능, 이산적 종료, 속성 보장)을 만족하면 어떤 도메인이든 적용 가능하다:

| 도메인 | 룰 예시 | 검증 방법 | 보장 속성 |
|---|---|---|---|
| API 설계 | 모든 엔드포인트에 인증 필수 | OpenAPI 스펙 검사 | 보안 커버리지 |
| 데이터베이스 | 모든 FK에 인덱스 존재 | 스키마 검사 | 쿼리 성능 하한 |
| 문서 | 모든 public API에 설명 존재 | 패턴 매칭 | 문서 완전성 |
| 인프라 | 모든 컨테이너에 리소스 제한 | YAML 검사 | 리소스 격리 |

### 6.2 제약 설계 원칙

좋은 수렴적 제약을 설계하려면:

1. **국소성**: 한 파일의 수정이 다른 파일의 위반 상태를 바꾸지 않아야 한다 → 단조 감소 보장
2. **독립성**: 룰 간 충돌이 없어야 한다 → 동시 충족 가능
3. **관측 가능성**: 위반이 구체적이어야 한다 → LLM이 무엇을 고칠지 알 수 있음
4. **의미성**: 위반 0건이 "그래서 뭐?"가 아닌 유의미한 속성을 보장해야 한다

filefunc의 22개 룰은 이 네 원칙을 모두 만족한다.

## 7. 논의

### 7.1 "모델이 아니라 문제를 구조화하라"

AI 업계의 현재 방향: 더 큰 모델, 더 긴 컨텍스트, 더 정교한 RAG. 전부 **읽기 시점의 해법**이다.

수렴적 제약은 반대 방향을 제안한다: **쓰기 시점에 구조를 강제하면, 읽기 시점의 문제가 사라진다.** 모델을 200B에서 2T로 키우는 대신, 코드를 구조화하여 20B 모델로도 충분하게 만든다.

이것은 소프트웨어 엔지니어링에서 수십 년간 반복된 교훈이다: gofmt가 포맷 논쟁을 제거하고, 타입 시스템이 런타임 에러를 컴파일 시점으로 옮기고, CI/CD가 "내 기계에서는 됐는데"를 제거했다. filefunc은 같은 원칙을 AI 에이전트 시대에 적용한다.

### 7.2 LLM을 시스템 안에 두다

기존 AI 코드 에이전트: LLM이 시스템 **밖에서** 코드를 통째로 읽고, 이해하고, 판단하고, 수정한다. 모든 인지 부하가 LLM에 집중된다.

수렴적 제약: LLM이 시스템 **안에서** 제약된 역할(위반 수정)만 수행한다. 판단은 기호 시스템이, 검증도 기호 시스템이, LLM은 변환만.

이것이 뉴로-심볼릭의 정의 그대로다: 기호 시스템이 문제를 정의하고 해를 검증하고, 신경망이 해를 생성한다.

### 7.3 수렴의 실용적 의미

수렴이 보장된다는 것은:

- **자동화 가능**: 사람 개입 없이 야간 배치로 실행 가능
- **재현 가능**: 같은 코드베이스에 같은 룰을 적용하면 같은 속성에 도달
- **측정 가능**: 진행도 = (초기 위반 - 현재 위반) / 초기 위반
- **위임 가능**: LLM의 판단력을 신뢰할 필요 없음, 결과만 검증하면 됨

## 8. 한계

- Go 전용 인스턴스. 다른 언어에는 다른 룰 설계가 필요하다.
- LLM이 특정 위반을 고치지 못하는 경우(예: 복잡한 아키텍처 변경) 수렴이 실패할 수 있다. 실험에서 실패 케이스를 보고한다.
- 수렴적 제약은 **구조적 속성**만 보장한다. 기능적 정확성은 `go test`로 별도 검증해야 한다.
- 10개 프로젝트 실험은 Go 생태계에 한정된다. 범용성 주장은 다른 도메인 적용 사례가 필요하다.
- 룰 설계의 품질은 주관적이다. "좋은 수렴적 제약"의 형식적 정의는 향후 연구 과제다.

## 9. 결론

LLM을 효과적인 코드 변환기로 만드는 방법은 모델을 키우는 것이 아니라, 모델이 풀어야 할 문제를 기계 검증 가능한 제약으로 정의하는 것이다.

수렴적 제약은 세 가지 조건으로 정의된다:

1. **기계 검증 가능** — LLM 없이 위반을 판정
2. **이산적 종료** — 위반 수 0이 명확한 종료 조건
3. **속성 보장** — 위반 0건이 증명 가능한 구조적 속성을 보장

filefunc은 이 방법론의 첫 번째 인스턴스로, 22개 룰과 `while violation > 0: fix` 루프만으로 임의의 Go 코드베이스를 Bohm-Jacopini 3분류가 성립하는 상태로 수렴시킨다. 10개 오픈소스 프로젝트에서의 실증은 수렴이 실용적으로 달성 가능하며, 수렴된 코드베이스에서 AI 에이전트의 성능이 향상됨을 보인다.

모델이 아니라 문제를 구조화하라. 그것이 뉴로-심볼릭 소프트웨어 엔지니어링의 핵심이다.
