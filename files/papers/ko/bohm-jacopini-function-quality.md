# 지배적 제어구조에 의한 함수 분류: Bohm-Jacopini의 코드 품질 룰 적용

## 초록

Bohm-Jacopini 정리(1966)는 순차(sequence), 선택(selection), 반복(iteration)으로 모든 계산 가능한 함수를 표현할 수 있음을 증명한다. 우리는 이 정리를 함수 수준에 적용한다: 네스팅 깊이 상한 2를 강제한 후, 각 함수 본문은 세 제어구조 중 정확히 하나가 지배한다. 이를 `control=` 어노테이션으로 형식화하고, AST 분석으로 검증하며, 획일적 라인 제한 대신 구조 인식 품질 기준을 정의하는 데 활용한다.

세 Go 프로젝트의 2,332개 함수(filefunc 136, whyso 84, fullend 1,055)에서 평가한 결과, 모든 함수가 모호한 경우 없이 정확히 세 범주 중 하나로 분류되었으며, 분포는 순차 44.1%, 반복 51.2%, 선택 4.7%였다. 깊이-2 강제가 필수 전제조건임을 확인했다: 이 없이는 함수가 혼합 제어구조를 보이며 깨끗한 분류를 거부한다.

## 1. 서론

### 1.1 획일적 품질 룰의 문제

정적 분석 도구는 획일적 임계값을 적용한다: 순환 복잡도 < 10, 함수 길이 < 100줄, 네스팅 깊이 < 4. 이 임계값은 모든 함수를 동일하게 취급하지만, 함수는 근본적으로 다른 구조를 갖는다:

- **순차** 함수는 12단계를 순서대로 호출한다(파이프라인). 150줄이지만 복잡도는 낮다 — 각 줄이 독립적.
- **선택** 함수는 10개 switch case를 디스패치하며 각 3~8줄. 119줄이지만 10개 파일로 쪼개면 전체 케이스를 한눈에 볼 수 없다.
- **반복** 함수는 5단계 AST를 순회한다. 깊이 5이지만 각 단계는 로직 없는 타입 단언 — 깊이가 데이터 차원을 반영할 뿐 복잡도가 아니다.

획일적 룰은 실제 복잡도를 놓치거나(너무 관대) 구조적 패턴을 벌한다(너무 엄격). 빠진 변수는 함수의 **지배적 제어구조**다.

### 1.2 함수 수준의 Bohm-Jacopini

Bohm-Jacopini 정리는 모든 계산 가능한 함수가 순차, 선택, 반복만으로 표현될 수 있다고 진술한다. 우리는 함수가 깊이 2로 제한될 때, 이 이론적 결과가 실천적으로 관찰 가능해짐을 발견한다: 각 함수 본문은 단일 제어구조가 **지배**한다.

이것은 정리의 재진술이 아니다. 정리는 표현력에 관한 것이고, 우리의 주장은 **분류 유용성**에 관한 것이다: 깊이가 제한된 함수는 단일 지배 제어구조를 보이며, 이것이 품질 특성을 예측한다.

### 1.3 선행 연구

우리가 아는 한, Bohm-Jacopini 분류를 개별 함수 본문에 품질 지표로 적용한 선행 연구는 없다. 관련 연구:

- **순환 복잡도** (McCabe, 1976): 결정 지점을 센다. 10개 case switch와 10단계 중첩 if-else를 구분하지 않는다.
- **인지 복잡도** (SonarSource, 2017): 네스팅에 페널티를 부여한다. 우리 접근에 가깝지만 함수를 구조적 범주로 분류하지 않는다.
- **Halstead 메트릭** (1977): 피연산자와 연산자를 측정한다. 내용 기반이지 구조 기반이 아니다.
- **ABC 메트릭**: 할당, 분기, 조건을 센다. 분류가 아닌 집계.

이 중 어느 것도 함수를 지배적 제어구조로 분류하지 않는다.

## 2. 방법

### 2.1 전제조건: 깊이-2 강제

먼저 엄격한 네스팅 깊이 상한을 강제한다:

| control | 깊이 상한 |
|---|---|
| sequence | 2 (깊이 1에 loop/switch 없음) |
| selection | 2 (깊이 1에 switch, 케이스 내부 로직은 깊이 2) |
| iteration | dimension + 1 (깊이 1에 loop, 본문은 깊이 2+) |

이 평탄화가 필수적이다. 제약 없는 코드에서 함수는 loop 안에 switch 안에 또 다른 loop을 포함할 수 있다 — 분류를 거부하는 혼합 구조. 깊이-2 강제는 각 함수가 최대 한 단계의 주요 제어구조만 갖게 하여, 지배 구조를 모호하지 않게 만든다.

### 2.2 분류 규칙

깊이-2 강제가 적용된 함수에 대해:

| 조건 | 분류 |
|---|---|
| 깊이 1에 `for`/`range` 문 존재 | **iteration** |
| 깊이 1에 `switch`/`type switch` 문 존재 | **selection** |
| 깊이 1에 loop도 switch도 없음 | **sequence** |

깊이 1의 early-return `if err != nil { return }`은 selection으로 분류하지 않는다 — 분기(선택)가 아니라 가드(탈출)이다.

### 2.3 AST 검증

`filefunc validate`는 `control=` 어노테이션과 실제 AST 간의 일관성을 강제한다:

| 룰 | 조건 | 심각도 |
|---|---|---|
| A9 | func 파일은 `control=` 필수 | ERROR |
| A10 | `control=selection`이지만 깊이 1에 switch 없음 | ERROR |
| A11 | `control=iteration`이지만 깊이 1에 loop 없음 | ERROR |
| A12 | `control=sequence`이지만 깊이 1에 loop/switch 존재 | ERROR |
| A13 | `control=selection`이지만 깊이 1에 loop 존재 | ERROR |
| A14 | `control=iteration`이지만 깊이 1에 switch 존재 | ERROR |

이 룰들은 분류를 주관적이 아닌 기계 검증 가능하게 만든다.

### 2.4 Dimension: 반복 깊이 세분화

반복 함수의 경우, 순회 대상 데이터가 자연스러운 네스팅 깊이를 결정한다. `dimension=N`을 도입한다:

- `dimension=1`: 단층 리스트 순회. 깊이 상한 = 2.
- `dimension=2`: 2차원 데이터 (예: paths -> operations). 깊이 상한 = 3. Named type 중첩 필수.
- `dimension=N`: N차원 데이터. 깊이 상한 = N + 1.

제약: dimension >= 2는 각 레벨에서 named type 모델링을 요구한다. Raw 중첩(`[][][]int`)은 금지 — 각 차원이 named type(struct 또는 interface)이어야 한다.

## 3. 깊이-2 조합 분석

깊이 1 요소가 loop, if, switch일 때, 9가지 깊이-2 조합이 가능하다:

| 깊이 1 -> 깊이 2 | 패턴 | 분류 | 빈도 |
|---|---|---|---|
| loop -> loop | 중첩 반복 | **iteration** | 드뭄 |
| loop -> if | 필터/선별 | **iteration** | 흔함 |
| loop -> switch | 순회+분기 | **iteration** | 흔함 |
| if -> loop | 조건부 반복 | **sequence** | 드뭄 |
| if -> if | 중첩 조건 | **sequence** | early return으로 대체 |
| if -> switch | 조건부 디스패치 | **sequence** | 드뭄 |
| switch -> loop | 케이스별 반복 | **selection** | 드뭄 |
| switch -> if | 케이스별 조건 | **selection** | 흔함 |
| switch -> switch | 중첩 분기 | **selection** | 드뭄 |

분류 규칙은 단순하다: 깊이 1이 범주를 결정한다. loop -> iteration, switch -> selection, if 또는 없음 -> sequence.

## 4. 구조 인식 품질 룰

### 4.1 차별화된 임계값

| control | Q1 (깊이) | Q3 (줄 수) | 근거 |
|---|---|---|---|
| sequence | <= 2 | 100 (WARNING), 200 (파이프라인) | 순차 단계; 길지만 단순 |
| selection | 케이스 내부 <= 2 | 300 (케이스 수에 비례) | 케이스를 한꺼번에 봐야 함 |
| iteration | <= dimension + 1 | 100 | 루프 본문이 복잡도의 핵심 |

### 4.2 획일적 룰이 실패하는 이유 (fullend Q3 데이터)

fullend의 Q3 위반 15건(>100줄)에서:

| 범주 | 건수 | 쪼개야 하나? |
|---|---|---|
| sequence (파이프라인/템플릿) | 9 | 4건 예, 5건 아니오 |
| selection (단층 디스패치) | 5 | 0건 예, 5건 아니오 |
| iteration (AST 워커) | 1 | 1건 예 |

**위반의 60%(9/15)가 100줄 이상이 적절하다.** 획일적 100줄 룰은 구조적으로 단순한 함수의 분리를 강제하여 가독성을 떨어뜨린다.

## 5. 평가

### 5.1 분류 완전성

| 프로젝트 | 함수 수 | sequence | selection | iteration | 분류 불가 |
|---|---|---|---|---|---|
| filefunc | 136 | 58 (42.6%) | 4 (2.9%) | 54 (39.7%) | 0 |
| whyso | 84 | 41 (48.8%) | 5 (6.0%) | 38 (45.2%) | 0 |
| fullend | 1,055 | 393 (37.3%) | 74 (7.0%) | 588 (55.7%) | 0 |
| **합계** | **1,275** | **492 (38.6%)** | **83 (6.5%)** | **680 (53.3%)** | **0** |

세 프로젝트 1,275개 함수에서 분류 불가 0건.

### 5.2 Dimension 분포 (반복 함수)

| dimension | 파일 수 | Q1 상한 | 의미 |
|---|---|---|---|
| 1 | 600 | 2 | 단층 리스트 순회 |
| 2 | 67 | 3 | 2차원 데이터 (paths->ops) |
| 3 | 10 | 4 | 3차원 데이터 (policies->rules->actions) |
| 4 | 2 | 5 | 4차원 데이터 (spec->existing 중복 제거) |
| 5 | 1 | 6 | 5차원 데이터 (AST 전체 순회) |

반복 함수의 88.2%가 dimension=1이며, 대부분의 반복이 단순 단층 순회임을 확인한다.

### 5.3 프로젝트 유형 가설

| 프로젝트 유형 | 예상 지배 구조 | 관측 (fullend) |
|---|---|---|
| 코드 생성기 | sequence | sequence 37.3% (파서가 iteration을 끌어올림) |
| CLI 도구 | sequence | (filefunc) sequence 42.6% |
| 데이터 처리기 | iteration | (whyso) iteration 45.2% |

분포가 프로젝트 목적과 상관관계를 보이며, 예측적 가치를 시사한다.

## 6. 논의

### 6.1 깊이-2가 전제조건인 이유

깊이-2 강제 없이 함수는 다음을 포함할 수 있다:
```go
for items {           // iteration
    switch item.Type { // selection
        case A:
            for sub {  // 또 iteration
            }
    }
}
```

이 함수는 반복인가? 선택인가? 둘 다? 깊이-2는 이를 반복 함수(외부 루프)와 선택 함수(switch 디스패치)로 분리하도록 강제하며, 각각이 분류 가능해진다.

### 6.2 AI 에이전트 시사점

`control=` 어노테이션은 읽기 전 전략을 가능케 한다:

- `control=selection`: 함수 전체를 한 번에 read. 부분 읽기는 케이스를 놓침.
- `control=iteration`: 루프 본문에 집중. 루프 외부는 초기화.
- `control=sequence`: 수정 대상 단계만 read. 나머지 단계는 what으로 충분.

AI 에이전트가 본문을 읽기 전에 어노테이션만으로 **어떻게 읽을지** 결정할 수 있다.

### 6.3 순환 복잡도와의 관계

순환 복잡도는 모든 결정 지점을 동등하게 센다. 10개 case switch는 CC=10으로, 10단계 중첩 if-else 체인과 동일한 점수다. 그러나 switch는 단층 디스패치(낮은 인지 부하)이고 중첩 if-else는 진짜 복잡하다. Bohm-Jacopini 분류는 이를 구분한다: switch는 selection(전체를 한 번에 읽기), 중첩 if-else는 깊이-2 강제 하에 여러 sequence 함수로 분리된다.

## 7. 한계

- Go 전용 평가. 다른 언어는 다른 깊이 상한이 필요할 수 있다 (예: Python의 컴프리헨션에 의한 암묵적 중첩).
- 한 개발자의 세 프로젝트. 오픈소스 코드베이스에 대한 광범위한 평가 필요.
- "분류 불가 0건" 결과는 깊이-2 강제에 의존한다. 이 제약 없는 프로젝트는 분류를 거부할 수 있다.
- Selection 함수가 드물다 (6.5%). selection 전용 품질 룰 검증에 더 많은 데이터 필요.

## 8. 결론

깊이-2 강제 후 함수 수준에 적용된 Bohm-Jacopini 정리는 세 가지 속성을 가진 실용적 분류를 산출한다:

1. **완전** — 1,275/1,275 함수 분류 (모호 0건)
2. **기계 검증 가능** — 6개 AST 룰이 어노테이션 일관성 검증
3. **품질 예측적** — 획일적 룰의 거짓 양성 60%를 회피하는 차별화된 임계값 가능

핵심 통찰: 깊이 강제는 단순한 품질 룰이 아니다 — **구조적 분류의 전제조건**이다. 평탄화된 함수가 지배적 제어구조를 드러내어 구조 인식 도구를 가능케 한다.
