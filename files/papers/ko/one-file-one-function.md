# 1파일 1함수: 파일 세분화가 AI 코드 에이전트 성능에 미치는 영향

## 초록

AI 코드 에이전트는 파일 단위로 코드를 읽는다. 파일에 함수가 20개 있으면, 하나의 함수를 읽기 위해 무관한 19개 함수를 컨텍스트 윈도우에 넣어야 한다 — 이를 **컨텍스트 오염**이라 부른다. 본 논문에서는 두 Go 프로젝트(21,232 LOC, 1,773 LOC)를 엄격한 "1파일 1함수" 컨벤션으로 리팩토링한 실증 데이터를 제시하며, 코드 구조 지표에 미치는 영향을 측정한다.

전환 후 평균 파일 크기가 244줄에서 25.4줄로 감소(-89.6%), 단일 책임 위반이 75.9%에서 0%로, 네스팅 깊이 위반이 148건 이상에서 0건으로 감소했다. 파일 세분화가 AI 코드 에이전트 효율의 일차적 지렛대임을 주장한다: 파일이 작을수록 읽기당 노이즈가 줄고, 파일 수준 메타데이터(이름, 어노테이션)가 자동으로 함수 수준 메타데이터가 된다.

## 1. 서론

### 1.1 읽기 단위 문제

AI 코드 에이전트(Claude Code, Cursor, Copilot)는 `read file` 연산으로 코드베이스를 탐색한다. 코드 검색의 원자 단위는 파일이다. 이것이 근본적 불일치를 만든다:

- **개발자 의도**: "함수 X를 이해해야 한다"
- **에이전트 행동**: "함수 X가 포함된 전체 파일을 읽어야 한다"
- **결과**: 함수 X는 30줄이지만 파일은 500줄

무관한 470줄은 공짜가 아니다. 연구가 이를 입증한다:

- **Lost in the Middle** (Liu et al., 2024): 긴 컨텍스트 중간의 관련 정보는 LLM 성능을 30% 이상 하락
- **Context Length Alone Hurts** (Amazon, 2025): 빈 패딩 토큰조차 성능을 13.9~85% 하락
- **Context Rot** (Chroma Research): 모든 테스트 모델에서 집중된 프롬프트가 전체 프롬프트를 능가

### 1.2 파일 세분화 명제

읽기 단위가 파일이면, **파일 세분화가 컨텍스트 품질을 결정한다**. 우리는 제안한다:

> 각 파일이 정확히 하나의 개념을 포함하게 하여 "에이전트가 필요한 것"과 "파일이 포함한 것" 사이의 간극을 최소화하라.

이것이 1파일 = 1함수(1F1F) 컨벤션이다.

## 2. 1F1F 컨벤션

### 2.1 룰

| 룰 | 설명 | 심각도 |
|---|---|---|
| F1 | 파일 하나에 함수 하나 (파일명 = 함수명) | ERROR |
| F2 | 파일 하나에 타입 하나 (파일명 = 타입명) | ERROR |
| F3 | 파일 하나에 메서드 하나 (receiver_method.go) | ERROR |
| F4 | init()은 func 또는 var와 함께만 허용 | ERROR |
| F5 | _test.go는 복수 함수 허용 | 예외 |
| F6 | 의미적 한 묶음인 const 허용 | 예외 |

### 2.2 네이밍 컨벤션

```
check_nesting_depth.go    → func CheckNestingDepth(...)
server.go                 → type Server struct{...}
server_start.go           → func (s *Server) Start(...)
```

파일명 = 개념명. 함수를 찾기 위해 grep이 필요 없다 — 파일 시스템이 곧 인덱스다.

### 2.3 어노테이션

각 파일은 기계 판독 가능한 메타데이터를 갖는다:

```go
//ff:func feature=validate type=rule control=iteration dimension=1
//ff:what Q1: 네스팅 깊이를 dimension 기반 상한과 대조하여 검증
func CheckNestingDepth(gf *model.GoFile) []model.Violation {
```

각 파일이 하나의 함수이므로, 파일 수준 어노테이션이 자동으로 함수 수준 어노테이션이 된다.

## 3. 사례 연구 1: fullend (21K LOC)

### 3.1 프로젝트 설명

fullend는 9개 단일 진실 소스(SSOT)의 정합성을 검증하고 코드를 생성하는 Go CLI다. 13단계(Phase031~043)에 걸쳐 전환.

### 3.2 구조 지표

| 지표 | Before | After | 변화 |
|---|---|---|---|
| Go 소스 파일 | 87 | 1,260 | +1,348% |
| 총 LOC | 21,232 | 31,976 | +50.6% |
| 평균 LOC/파일 | 244 | 25.4 | -89.6% |
| 중앙값 LOC/파일 | — | 21 | |
| 함수 수 | 617 | 1,077 | +460 (추출된 헬퍼) |
| 타입 수 | 41 | 218 | +177 (분리된 내부 타입) |
| 평균 func/파일 | 7.0 | 0.85 | -87.9% |
| 최대 파일 크기 | 1,113줄 | 232줄 | -79.2% |

### 3.3 LOC 증가 분석

+10,744 LOC 증가는 전부 구조적 오버헤드:

| 출처 | 줄 수 |
|---|---|
| 어노테이션 (파일당 //ff:func + //ff:what) | ~2,500 |
| 파일당 package 선언 + import 블록 | ~5,000 |
| 추출된 헬퍼 함수 시그니처 + import | ~3,200 |

순수 로직 LOC 증가 0. 모든 추가 줄은 파일 분리에 의한 보일러플레이트.

### 3.4 파일 크기 분포

| 구간 | Before | After |
|---|---|---|
| 1~25줄 | 3.4% | 64.8% |
| 26~50줄 | 14.9% | 27.2% |
| 51~100줄 | 16.1% | 7.0% |
| 101~200줄 | 34.5% | 0.8% |
| 201줄+ | 31.0% | 0.2% |

**전환 후 99.0%의 파일이 100줄 이하.**

### 3.5 SRP 위반

Before: 87개 파일 중 66개(75.9%)가 F1 위반(파일에 복수 함수). 최악: symbol.go에 35개 함수 혼재.

After: 1,260개 파일 중 F1 위반 0건. 모든 파일이 정확히 하나의 함수, 타입, 또는 메서드를 포함.

### 3.6 네스팅 깊이

13단계에 걸쳐 148건 이상의 Q1(깊이) 위반이 0건으로 해소. 깊이-2 강제가 헬퍼 함수 추출을 유도했고, 이것이 다시 더 많은 단일 함수 파일을 생성했다.

## 4. 사례 연구 2: whyso (1.7K LOC)

### 4.1 프로젝트 설명

whyso는 Claude Code 세션에서 파일 변경 이력을 추출하고 tree-sitter로 키워드 맵을 생성하는 Go CLI다.

### 4.2 구조 지표

| 지표 | Before | After | 변화 |
|---|---|---|---|
| Go 소스 파일 | 12 | 99 | +725% |
| 총 LOC | 1,773 | 2,417 | +36.3% |
| 평균 LOC/파일 | 147.8 | 24.4 | -83.5% |
| 함수 수 | 65 | 84 | +19 헬퍼 |
| 평균 func/파일 | 5.4 | 1.0 | -81.5% |
| 최대 파일 크기 | 410줄 | 109줄 | -73.4% |

### 4.3 전환 비용

| 지표 | 값 |
|---|---|
| 전환 시간 | ~5분 (AI 에이전트 6개 병렬) |
| LOC 증가 | +644줄 (+36.3%) |
| 어노테이션 오버헤드 | 192줄 (총 LOC의 7.9%) |

### 4.4 깊이 해소

23건의 Q1 위반을 다음 기법으로 해소:

| 기법 | 건수 | 새 파일 필요 |
|---|---|---|
| 조건 병합 | 5 | 0 |
| early continue/return | 5 | 0 |
| 헬퍼 함수 추출 | 13 | +13파일 |

## 5. 컨텍스트 오염 분석

### 5.1 이론 모델

N개 함수가 F개 파일에 분산된 프로젝트에서, 읽기당 평균 컨텍스트 오염:

```
오염률 = (N/F - 1) / (N/F) = 1 - F/N
```

| 프로젝트 | Before (F/N) | After (F/N) | Before 오염 | After 오염 |
|---|---|---|---|---|
| fullend | 87/617 = 0.14 | 1,260/1,077 = 1.17 | 86% | 0% |
| whyso | 12/65 = 0.18 | 99/84 = 1.18 | 82% | 0% |

전환 전에는 함수 하나를 읽으면 평균 5~7개 무관한 함수가 딸려온다. 전환 후에는 정확히 0개.

### 5.2 토큰 낭비 추정

전환 후 평균 함수 크기 30토큰 가정:

| 프로젝트 | Before: 읽기당 낭비 토큰 | After: 읽기당 낭비 토큰 |
|---|---|---|
| fullend | 30 * (7.0 - 1) = 180토큰 | 0토큰 |
| whyso | 30 * (5.4 - 1) = 132토큰 | 0토큰 |

일반적인 Claude Code 세션에서 50개 파일을 읽을 때, 이는 6,600~9,000토큰의 노이즈를 절약한다 — "Lost in the Middle" 연구에 따르면 모델 성능을 하락시키는 토큰들.

## 6. 인덱스로서의 파일 시스템

### 6.1 1F1F 이전

함수 `CheckNestingDepth`를 찾으려면:
```
1. grep -r "func CheckNestingDepth" .     → 파일 찾기
2. read validate.go                        → 500줄, 20개 함수
3. 파일 내에서 함수 위치 탐색               → 스크롤/검색
```

### 6.2 1F1F 이후

```
1. read check_nesting_depth.go            → 25줄, 1개 함수
```

파일 시스템 자체가 검색 가능한 인덱스가 된다. 파일명 = 함수명. 정확 매칭에 grep 불필요. `ls internal/validate/`가 모든 검증 룰을 보여준다.

### 6.3 Git 통합

파일 수준 git 연산이 함수 수준 연산이 된다:

- `git log check_nesting_depth.go` = 함수 변경 이력
- `git blame check_nesting_depth.go` = 함수 작성자
- `git diff check_nesting_depth.go` = 함수 diff

hunk 분석 불필요. 전체 파일 diff가 곧 함수 diff.

## 7. 비용과 트레이드오프

### 7.1 파일 수 폭발

87 -> 1,260파일(+1,348%)이 주요 비용. 영향:

- **IDE 탐색**: 파일명 = 함수명 컨벤션으로 완화
- **PR 리뷰**: PR당 더 많은 파일 변경. 파일당 더 작은 diff로 완화.
- **빌드 시스템**: Go는 패키지 단위로 컴파일, 파일 단위 아님. 빌드 영향 없음.

### 7.2 LOC 오버헤드

fullend에서 +50.6% LOC 증가는 구조적 보일러플레이트. 이것이 구조의 대가다. 각 파일은 자체 package 선언, import, 어노테이션이 필요하다.

오버헤드 비율은 함수 복잡도가 높을수록 감소한다: 5줄 함수는 자체 파일에서 300% 오버헤드(15줄 보일러플레이트)이지만, 50줄 함수는 30% 오버헤드.

### 7.3 Import 중복

같은 import가 패키지 내 50개 파일에 나타날 수 있다. 이것은 텍스트 중복이지 논리적 중복이 아니다. Go 빌드 시스템은 컴파일 시점에 중복을 제거한다.

## 8. 기존 관행과 비교

| 접근 | 파일 세분화 | 인덱스 방법 | AI 에이전트 비용 |
|---|---|---|---|
| 표준 Go | 200~500 LOC/파일 | grep + read | 높은 오염 |
| Java (1클래스/파일) | 100~300 LOC/파일 | 클래스명 | 중간 오염 |
| **1F1F** | **20~30 LOC/파일** | **파일명** | **오염 0** |
| 마이크로서비스 | 해당 없음 (서비스 수준) | API 디스커버리 | 다른 범위 |

1F1F는 Java가 1클래스/파일 룰로 부분적으로 구현한 "파일당 1개념" 원칙의 논리적 극한이다.

## 9. 한계

- 한 개발자의 두 Go 프로젝트. 다국어, 다팀 평가 필요.
- AI 에이전트 태스크 완수 정확도(전/후)의 직접 측정 없음. 구조적 대리 지표를 측정.
- "Lost in the Middle" 효과는 모델 의존적. 미래 모델은 컨텍스트 노이즈에 덜 민감할 수 있음.
- 파일 수 폭발은 매우 큰 규모(10만+ 파일)에서 실용적 한계가 있을 수 있음.

## 10. 결론

파일 세분화는 AI 코드 에이전트 컨텍스트 품질의 일차적 지렛대다. 1F1F 컨벤션은 컨텍스트 오염 — 하나의 함수를 찾기 위해 파일을 읽을 때 에이전트의 컨텍스트 윈도우에 들어오는 무관한 코드 — 을 제거한다.

총 23,005 LOC의 두 프로젝트에서:

1. **컨텍스트 오염: 82~86% -> 0%** — 모든 읽기가 정확히 필요한 것만 반환
2. **평균 파일 크기: 147~244줄 -> 24~25줄** — 84~90% 감소
3. **SRP 위반: 76~100% -> 0%** — 구조적 품질 보장
4. **깊이 위반: 23~148+ -> 0** — 복잡도 제한

비용 — 36~51% LOC 증가와 725~1,348% 파일 수 증가 — 은 오염 없는 읽기를 사는 구조적 오버헤드다. 읽기가 쓰기보다 압도적으로 많은 AI 보조 개발에서, 이 트레이드오프는 구조 쪽으로 기운다.
