# 1만 함수를 30K 토큰에: LLM 코드 탐색 진입점으로서의 압축 심볼 맵

## 초록

LLM 코드 에이전트는 특정 파일로 탐색하기 전에 코드베이스의 전체적인 조감이 필요하다. 현재 접근은 프로젝트 구조 전체를 덤프하거나(비쌈) 벡터 검색에 의존한다(손실 있음). 우리는 **심볼 맵** — `[패키지]함수1,함수2,...` 표기법으로 코드베이스의 모든 함수명을 인코딩하는 압축 표현 형식 — 을 제시한다. 함수당 약 3토큰을 달성한다.

1만 함수 규모의 코드베이스가 약 30K 토큰에 들어간다 — 모든 LLM 프롬프트에 포함할 수 있을 만큼 작다. 호출 그래프 순회 도구와 결합하면, **맵으로 진입점, 체인으로 영향 범위**라는 2단계 탐색 패턴이 만들어진다. 세 Go 프로젝트에서 토큰 효율을 측정하고, 코드북 기반 탐색 및 전체 프로젝트 덤프와 비교한다.

## 1. 서론

### 1.1 콜드 스타트 문제

AI 코드 에이전트가 태스크를 시작할 때, 콜드 스타트에 직면한다: 어떤 파일이 관련 있는가? 현재 접근:

| 접근 | 토큰 비용 | 정확도 | 지연 |
|---|---|---|---|
| 전체 프로젝트 덤프 | O(전체 LOC) | 완벽 | 비쌈 |
| RAG / 벡터 검색 | O(쿼리 + k개 결과) | 근사적 | 인덱스 필요 |
| 사용자 지정 파일 | O(지정된 만큼) | 사용자 의존 | 수동 |
| grep + read | O(매칭 * 파일크기) | 정확한 용어에 좋음 | 여러 라운드 |

저렴하고, 완전하고, 정확한 전역 뷰를 제공하는 것이 없다.

### 1.2 심볼 맵 접근

심볼 맵은 프로젝트의 모든 함수/타입을 패키지별로 묶어, 최소한의 문법으로 나열한다:

```
# whyso/v1

## go
[codemap]BuildMap,FormatMap,NeedsUpdate,buildSections,...
[history]BuildHistories,BuildHistoriesIncremental,BuildIndex,...
[main]clearCache,formatChangeRow,getSessionsDir,main,...
[model]ContentBlocks,IsUserMessage,UserContent
[output]FormatJSON,FormatYAML,OutputPath,ReadYAML,...
[parser]DetectSessionsDir,ExtractChanges,ListSessions,...
```

이것이 whyso의 완전한 심볼 맵: ~80개 함수가 ~200토큰. 핵심 트레이드오프: **함수당 메타데이터 없음**(설명, 타입, 관계 없음) — 이름만, 패키지로 그루핑.

### 1.3 메타데이터를 넣지 않는 이유

메타데이터(설명, 입출력 타입)를 추가하면 토큰이 선형으로 증가한다:

| 형식 | 함수당 토큰 | 1만 함수 |
|---|---|---|
| 이름만 (맵) | ~3 | ~30K |
| 이름 + 1줄 설명 | ~15 | ~150K |
| 이름 + 타입 + 설명 | ~30 | ~300K |
| 전체 함수 시그니처 | ~50 | ~500K |

~30K 토큰이면 심볼 맵이 어떤 LLM 컨텍스트에든 편안하게 들어간다. 150K 이상이면 실제 코드와 컨텍스트 공간을 경쟁한다. 설계 선택은 의도적이다: **함수 이름만으로도 LLM 탐색에 충분하다. LLM이 네이밍 컨벤션에서 목적을 유추하기 때문이다**.

## 2. 설계

### 2.1 형식 명세

```
# <도구>/v1

## <언어>
[<패키지>]<함수1>,<함수2>,...,<함수N>
```

규칙:
- 패키지 내 함수는 쉼표로 구분, 한 줄에 표현
- 패키지명이 그룹 헤더 역할
- 언어 섹션으로 다국어 프로젝트 지원
- 버전 헤더로 형식 발전 가능

### 2.2 생성

whyso는 tree-sitter로 지원 언어에서 심볼을 추출한다:

| 언어 | 추출 심볼 |
|---|---|
| Go | 함수, 메서드, 타입 |
| TypeScript/JavaScript | 함수, 클래스, export |
| Python | 함수, 클래스 |
| Rust | 함수, struct, trait |
| OpenAPI | Operation ID, 경로 |
| SQL | 테이블명, 함수명 |
| Rego | 룰명 |
| SSaC | 서비스 함수 |

생성은 증분적: 마지막 실행 이후 수정된 파일만 재파싱 (`mtime` 비교).

### 2.3 토큰 효율

압축 형식이 높은 정보 밀도를 달성한다:

```
[codemap]BuildMap,FormatMap,NeedsUpdate,buildSections,collectCaptures,dedupe,...
```

토큰화 (cl100k_base):
- `[codemap]` → 3토큰
- 각 함수명 → 1~3토큰 (camelCase가 예측 가능하게 분리)
- 쉼표 → 인접 토큰과 공유
- 평균: 함수당 ~3토큰

### 2.4 진부화 관리

맵은 `mtime` 기반 캐싱을 사용한다:
- `whyso map`이 소스 파일 수정 시각을 `.whyso/_map.md`와 비교
- 변경된 파일만 재파싱
- `whyso map -f`로 강제 재생성
- 변경 없으면 "up to date" 출력

## 3. 2단계 탐색: 맵 + 체인

### 3.1 패턴

```
1단계: whyso map → "어디에 뭐가 있지?"
  LLM이 심볼 맵 읽음 (~30K 토큰)
  이름으로 진입점 함수 식별

2단계: filefunc chain <함수> → "뭐에 영향 주지?"
  진입점에서 호출 그래프 순회
  호출자, 피호출자, 공동 호출 함수를 //ff:what 설명과 함께 반환
```

### 3.2 상호 보완적 역할

| 도구 | 제공하는 것 | 토큰 비용 | 범위 |
|---|---|---|---|
| whyso map | 전역 인덱스 (모든 함수) | ~30K 고정 | 프로젝트 전체 |
| filefunc chain | 로컬 그래프 (호출 관계) | ~500/쿼리 | 함수 이웃 |
| codebook | 도메인 어휘 (feature, type) | ~200 고정 | 분류 |

맵은 "뭐가 있나?"에 답한다. 체인은 "뭐와 연결되나?"에 답한다. 코드북은 "어떻게 분류되나?"에 답한다.

### 3.3 워크플로우 예시

사용자 요청: "이력 병합 버그 수정해"

```
1. LLM이 심볼 맵 읽음
   → [history]BuildHistories,BuildHistoriesIncremental,Merge,... 발견
   → Merge를 타겟으로 식별

2. filefunc chain func Merge --chon 2 --meta what
   → Merge (what="같은 파일에 대한 겹치는 이력 항목을 병합")
     1촌 calls: entryKey, lastEntry
     1촌 called-by: BuildHistories, BuildHistoriesIncremental
     2촌 co-called: processChanges, skipBySince

3. LLM이 읽기: merge.go, entry_key.go, last_entry.go
   → 3파일, 총 ~75줄, 노이즈 0
```

맵 없이는 에이전트가 "merge"를 grep하여 패키지 간 거짓 양성을 만날 수 있다. 맵은 즉각적이고 모호하지 않은 함수 위치를 제공한다.

## 4. 대안과 비교

### 4.1 코드북 기반 탐색 대비

filefunc의 코드북 + `filefunc context` 파이프라인:
```
코드북 (250토큰) → LLM이 feature 선택 → grep feature=X → what 읽기 → body 읽기
```

심볼 맵:
```
맵 (30K 토큰) → LLM이 함수 식별 → chain → body 읽기
```

| 측면 | 코드북 | 심볼 맵 |
|---|---|---|
| 토큰 비용 | 250 (코드북) + LLM 호출 | 30K (맵) + 선택에 LLM 호출 0 |
| 선택에 LLM 필요 | 예 (feature 선택) | 아니오 (LLM이 맵을 직접 읽음) |
| 세분화 | Feature 수준 (10~50파일) | 함수 수준 (1파일) |
| 설정 비용 | 수동 코드북 설계 | 자동 (tree-sitter) |

코드북이 대형 프로젝트에서 토큰 효율적(250 vs 30K 토큰)이지만 수동 어휘 설계 필요. 심볼 맵은 자동이지만 토큰을 더 소비.

**최적 조합**: 코드북은 도메인 분류에, 심볼 맵은 함수 수준 식별에. 서로 다른 세분화 수준에서 작동하며 경쟁하지 않는다.

### 4.2 전체 프로젝트 덤프 대비

전체 프로젝트 덤프는 모든 소스 코드를 포함:

| 프로젝트 | 전체 덤프 토큰 | 심볼 맵 토큰 | 비율 |
|---|---|---|---|
| filefunc (3K LOC) | ~12,000 | ~500 | 24배 |
| whyso (2.4K LOC) | ~10,000 | ~300 | 33배 |
| fullend (32K LOC) | ~130,000 | ~4,000 | 32배 |

심볼 맵은 전체 덤프보다 ~30배 압축적이면서 완전한 함수 목록을 유지한다.

### 4.3 RAG / 벡터 검색 대비

| 측면 | RAG | 심볼 맵 |
|---|---|---|
| 인프라 | 임베딩 모델 + 벡터 DB | 디스크의 파일 |
| 인덱스 크기 | O(청크 * 차원) 부동소수점 | O(함수) 토큰 |
| 쿼리 | 근사 최근접 이웃 | 정확한 문자열 검색 |
| 결과 유형 | 순위 목록 (누락 가능) | 완전한 목록 |
| 갱신 비용 | 변경 청크 재임베딩 | 변경 파일 재파싱 |

RAG는 관련 청크를 검색하지만 비직관적 이름의 함수를 놓칠 수 있다. 심볼 맵은 완전한 목록을 제공한다 — 숨겨진 것이 없다.

## 5. 평가

### 5.1 토큰 측정

| 프로젝트 | 함수 | 타입 | 맵 토큰 | 토큰/심볼 |
|---|---|---|---|---|
| filefunc | 136 | 15 | ~500 | 3.3 |
| whyso | 84 | 12 | ~300 | 3.1 |
| fullend | 1,077 | 218 | ~4,000 | 3.1 |

다른 규모의 프로젝트에서 일관되게 심볼당 ~3토큰.

### 5.2 스케일링 전망

| 프로젝트 규모 | 함수 수 | 추정 맵 토큰 | 200K 컨텍스트 대비 % |
|---|---|---|---|
| 소형 (5K LOC) | 200 | 600 | 0.3% |
| 중형 (30K LOC) | 1,200 | 3,600 | 1.8% |
| 대형 (100K LOC) | 5,000 | 15,000 | 7.5% |
| 초대형 (500K LOC) | 10,000 | 30,000 | 15% |
| 극대형 (1M LOC) | 20,000 | 60,000 | 30% |

500K LOC(1만 함수)까지 맵이 200K 컨텍스트 윈도우의 15% 미만을 소비. 그 이상은 계층적 그루핑이나 온디맨드 로딩이 필요할 수 있다.

### 5.3 이름의 정보성

심볼 맵은 함수 이름이 정보적임에 의존한다. 네이밍 컨벤션(filefunc의 snake_case 파일 = camelCase 함수)을 따르는 프로젝트에서:

```
[validate]CheckNestingDepth,CheckOneFileOneFunc,CheckCodebookValues,...
```

이 이름을 읽는 LLM은:
- `CheckNestingDepth`를 "네스팅 깊이" 쿼리와 관련있다고 식별 가능
- `validate` 패키지가 검증 룰을 포함한다고 유추 가능
- `CheckOneFileOneFunc`과 `CheckOneFileOneType`을 설명 없이 구분 가능

나쁜 네이밍(`doStuff`, `helper1`, `process`)의 프로젝트에서는 심볼 맵이 열화된다. **맵 품질 = 네이밍 품질**.

## 6. 한계

- 토큰 효율은 네이밍 컨벤션에 의존. 나쁜 이름의 함수는 탐색 정확도를 낮춤.
- 메타데이터가 없으므로 LLM이 이름만으로 함수 목적을 유추해야 함. 모호한 이름의 경우 chain의 `--meta what`이 2단계에서 설명을 제공.
- 2만 함수(6만 토큰)를 넘으면 계층적 맵이나 온디맨드 로딩이 필요할 수 있음.
- 현재 Go 프로젝트에서만 측정. 다른 언어는 다른 토큰/심볼 비율을 가질 수 있음.
- 심볼 맵을 사용한 LLM 탐색의 정식 정확도 평가 없음 (향후 연구 예정).

## 7. 향후 연구

- **정확도 벤치마크**: 컨텍스트에 심볼 맵 유무에 따른 LLM 태스크 완수 정확도 측정
- **계층적 맵**: 매우 큰 프로젝트를 위한 패키지 수준 요약 + 드릴다운
- **프로젝트 간 맵**: 의존성 라이브러리의 심볼 맵 (filefunc 레지스트리 개념과 유사)
- **점증적 컨텍스트**: 맵으로 시작, 필요에 따라 chain 결과와 파일 내용을 점진적으로 추가

## 8. 결론

`[패키지]함수1,함수2,...`를 인코딩하는 심볼 맵은 함수당 ~3토큰을 달성하여, 1만 함수가 30K 토큰에 들어간다. 이것은 LLM 코드 에이전트에게 저렴하고, 완전하고, 항상 사용 가능한 코드베이스 전역 인덱스를 제공한다.

호출 그래프 순회(`filefunc chain`)와 결합한 2단계 패턴 — **맵으로 진입, 체인으로 범위** — 은 벡터 검색 인프라, 수동 코드북 설계, 비싼 프로젝트 덤프 없이 정밀한 탐색을 가능케 한다.

핵심 통찰: **함수 이름이 곧 메타데이터다**. 잘 명명된 코드베이스에서, 이름의 목록만으로 LLM이 탐색하기에 충분하다. 심볼 맵은 이 목록을 모든 프롬프트에 포함할 수 있을 만큼 압축한다.
