# 코드베이스 탐색에 250토큰이면 충분하다: LLM 에이전트를 위한 신경-기호 코드 탐색

## 초록

AI 코드 에이전트는 턴당 30만~100만 토큰의 컨텍스트를 처리하지만, 그 중 1% 미만만이 태스크와 관련이 있다. 본 논문에서는 읽기 시점의 검색이 아닌 쓰기 시점의 구조화를 통해 코드 이해에 필요한 연산량을 126,000배 — 140페타FLOP에서 1.1테라FLOP으로 — 줄이는 신경-기호 코드 탐색 시스템 filefunc을 제시한다.

이 시스템은 기계 검증 가능한 어노테이션과 함께 "1파일 1개념" 컨벤션을 강제한 뒤, 3단계 파이프라인을 사용한다: (1) 8B 모델이 250토큰으로 코드북에서 관련 코드 도메인을 선택하고, (2) 20B 로컬 모델이 함수 설명을 스코어링하고, (3) 같은 20B 모델이 함수 본문을 스코어링한다. 프론티어 모델은 실제 필요한 960토큰만 받는다.

두 Go 프로젝트(filefunc 자체와 whyso)에서 실험한 결과, 총 5,760토큰의 LLM 호출 3회로 다양한 코드 탐색 쿼리에서 4/5 정확도를 달성했다 — 프론티어 모델에 12,000 프로젝트 토큰 또는 350,000 컨텍스트 토큰을 통째로 넣는 방식 대비.

## 1. 문제

### 1.1 컨텍스트 윈도우 세금

현대 AI 코드 에이전트(Claude Code, Cursor, Copilot)는 20만~100만 토큰의 컨텍스트 윈도우 내에서 동작한다. 일반적인 Claude Code 세션은 30만~40만 토큰의 누적 컨텍스트를 운반한다. 200B 파라미터 모델의 경우, 턴당 비용은:

```
2 × 200B × 350,000 = 140,000,000 TFLOPs
```

그러나 사용자가 "네스팅 깊이 검증 로직을 수정하라"고 요청하면, 관련 함수는 3~4개(~960토큰)에 불과하다. 나머지 349,000토큰은 모델이 처리해야 하는 노이즈다.

### 1.2 검색의 역전

업계의 대응은 검색 증강 생성(RAG)이었다: 코드를 벡터로 임베딩하고, ANN 인덱스를 구축하고, 유사 청크를 검색한다. 이 접근은:

- 구조를 파괴한다 (코드 → 부동소수점 벡터 → 근사 최근접 이웃)
- 비싼 임베딩 모델과 벡터 데이터베이스를 필요로 한다
- 유사도로 순위를 매긴 근사 결과를 산출한다 — 분류로 필터링한 정확한 결과가 아니라

우리는 이것이 거꾸로라고 주장한다. 문제는 "비정형 코드를 더 잘 검색하는 방법"이 아니라 "코드가 애초에 왜 비정형인가"다.

## 2. 접근: 쓰기 시점 구조화

### 2.1 filefunc 컨벤션

모든 Go 소스 파일은 정확히 하나의 함수 또는 타입을 포함한다 (1파일 = 1개념). 각 파일은 기계 검증 가능한 어노테이션을 갖는다:

```go
//ff:func feature=validate type=rule control=sequence
//ff:what Q1: 네스팅 깊이를 dimension 기반 상한과 대조하여 검증
func CheckNestingDepth(gf *model.GoFile) []model.Violation {
```

프로젝트 수준 코드북이 허용 어휘를 정의한다:

```yaml
required:
  feature: [validate, annotate, chain, parse, codebook, report, cli, context]
  type: [command, rule, parser, walker, model, formatter, loader, util]
```

### 2.2 검색 인덱스로서의 코드북

코드북은 문서가 아니다 — 검색 인덱스다. 각 `feature=X` 어노테이션은 코드베이스를 상호 배타적 집합으로 분할한다. 하나의 feature 값으로 70~99%의 파일을 제거한다:

| feature | 파일 수 | 전체 대비 % |
|---|---|---|
| validate | 35 | 28% |
| parse | 37 | 30% |
| chain | 28 | 22% |
| cli | 12 | 10% |
| codebook | 1 | 1% |

이것은 순위 매김이 아니라 분류다. 임베딩 없음. 유사도 점수 없음. `feature=validate` 문자열 매칭은 정확한 결과를 반환하는 집합 연산이다.

### 2.3 SILK 방법론

filefunc은 소스 코드에 적용된 SILK(Symbolic Index for LLM Knowledge) 아키텍처를 구현한다:

| SILK 원칙 | filefunc 구현 |
|---|---|
| 쓰기 시점 구조화 | `//ff:func feature=X type=Y` 어노테이션 |
| 코드북 어휘 제어 | `codebook.yaml` + A2 검증 |
| 기호 필터 → 신경 정제 | Feature 필터 → LLM what/body 스코어링 |
| VALID 기계 검증 | `filefunc validate` (22개 룰) |
| 순위 매김 대신 분류 | Feature 집합 연산, 벡터 유사도 아님 |

## 3. 파이프라인

### 3.1 4단계 컨텍스트 파이프라인

```
입력: 사용자 프롬프트 (자연어)

1단계 — Feature 선택 (8B 모델, 250토큰, API 호출 1회)
  코드북 + 프롬프트 → LLM이 관련 feature 1~2개 선택
  "네스팅 깊이 수정" → ["validate"]

2단계 — Feature 필터 (정적, 0 FLOPs)
  feature 어노테이션으로 프로젝트 파일 필터링
  161파일 → 35파일

3단계 — What 스코어링 (20B 로컬 모델, ~3,450토큰, 35회 호출)
  각 함수의 //ff:what을 프롬프트와 대조 스코어링
  35파일 → 6파일 (rate >= 0.2)

4단계 — Body 스코어링 (20B 로컬 모델, ~1,100토큰, 6회 호출)
  각 함수 본문을 프롬프트와 대조 스코어링
  6파일 → 3파일 (rate >= 0.5)

출력: 3개 함수, Go 소스 960토큰
```

총 LLM 연산량: 1,114 TFLOPs. 프론티어 모델은 960토큰만 수신.

### 3.2 연산량 비교

| 방법 | 모델 | 토큰 | FLOPs |
|---|---|---|---|
| 순진한 컨텍스트 | Opus 200B | 350,000 | 140,000,000 T |
| 프로젝트 덤프 | Opus 200B | 12,000 | 4,800 T |
| **filefunc 파이프라인** | **8B + 20B + Opus** | **250 + 4,550 + 960** | **1,114 T** |
| **감소율** | | | **126,000배** |

### 3.3 비용

| 단계 | 모델 | 비용 |
|---|---|---|
| Feature 선택 | Haiku API | $0.0003 |
| What + Body 스코어링 | 로컬 20B | 전기료만 |
| 최종 답변 | Opus | 350K 대신 960토큰 |

## 4. 실험

### 4.1 설정

- 프로젝트: filefunc (161파일, 3,037 LOC), whyso (99파일, 2,417 LOC)
- Feature 선택 모델: gpt-oss:20b (ollama 로컬)
- 스코어링 모델: gpt-oss:20b (ollama 로컬)
- 검증: `filefunc validate` — 두 프로젝트 모두 위반 0건

### 4.2 코드 탐색 정확도

| 쿼리 | 선택된 Feature | 상위 결과 | 정답? |
|---|---|---|---|
| "네스팅 깊이 검증 수정" | validate | CheckNestingDepth [0.80], depthLimit [0.80] | O |
| "코드북 파싱 변경" | parse | ParseCodebook [1.00] | O |
| "chain 출력 포맷 변경" | chain | FormatChain [0.90], formatMeta [0.55] | O |
| "LLM what-body 검증 수정" | validate (오답) | (결과 없음) | X |
| "어노테이션 파싱 수정" | parse | ParseAnnotation [0.90], ApplyAnnotationLine [0.80] | O |

4/5 정확도. 실패 케이스("LLM 검증")는 20B 모델이 "검증"을 `validate`로 매핑한 feature 선택 오류. 8B+ API 모델(Haiku, Gemini Flash)로 해결 가능.

### 4.3 Reranker 비교

vLLM + Qwen3-Reranker (0.6B, 4B)를 cross-encoder 스코어러로도 테스트:

| 방법 | 정확도 | 문제 |
|---|---|---|
| Reranker 0.6B | 0/5 | 점수가 평탄 (0.1~0.2) |
| Reranker 4B | 0/5 | 점수가 전부 높음 (0.6~0.96), 구분 불가 |
| **코드북 + LLM 스코어링** | **4/5** | Feature 선택이 병목 |

Reranker는 같은 패키지 내 모든 함수가 코드 쿼리에 "관련"되기 때문에 실패한다 — 도메인 내 구분이 불가능하다. 코드북 분류는 먼저 분할함으로써 이를 해결한다.

### 4.4 filefunc 도입 지표 (whyso)

| 지표 | Before | After |
|---|---|---|
| 파일 수 | 12 | 99 |
| 평균 LOC/파일 | 147.8 | 24.4 |
| SRP 위반 | 12/12 (100%) | 0/99 (0%) |
| Depth 위반 | 23 | 0 |
| 어노테이션 커버리지 | 0% | 100% |
| 전환 시간 | — | ~5분 (AI 에이전트 6개 병렬) |

## 5. 250토큰 탐색 명제

8B 모델에 대한 250토큰 코드북 쿼리는 200B 모델이 350,000토큰을 처리해야 할 코드베이스를 탐색하기에 충분하다. 이것은 최적화가 아니다 — 단일 설계 원칙에서 도출되는 126,000배 연산량 감소다:

**쓰기 시점에 코드를 구조화하면, 읽기 시점의 검색은 공짜가 된다.**

구조화 비용(어노테이션, 코드북, 검증)은 파일 쓰기당 한 번 지불된다. 검색 비용(350K 토큰 x 200B 파라미터)은 읽기마다 지불된다. 읽기가 쓰기보다 압도적으로 많은 AI 보조 개발에서, 구조화가 항상 더 저렴하다.

### 5.1 광범위한 시사점

이 결과는 도메인에 무관하다. LLM이 긴 컨텍스트를 처리하는 모든 곳에 같은 원칙이 적용된다:

- **문서**: 주제 코드북으로 섹션에 어노테이션 → 8B가 관련 섹션 선택
- **대화**: 의도 코드북으로 메시지에 태그 → 컨텍스트 포함 전 필터링
- **지식베이스**: SILK의 64비트 구조화 식별자 → bit-AND 필터링

패턴은 항상 동일하다: 소형 모델 + 코드북 → 분류 → 프론티어 모델은 중요한 것만 수신.

## 6. 관련 연구

- **RAG** (Lewis et al., 2020): 벡터 유사도로 검색. filefunc은 기호 분류로 검색.
- **Lost in the Middle** (Liu et al., 2024): 컨텍스트 중간의 관련 정보는 성능을 30%+ 하락. filefunc은 중간을 제거.
- **Context Rot** (Chroma Research): 집중된 프롬프트 > 전체 프롬프트. filefunc은 집중된 프롬프트를 자동 생성.
- **SILK** (Park, 2026): 지식을 위한 신경-기호 검색 아키텍처. filefunc은 코드에 적용된 SILK.

## 7. 한계

- Feature 선택 정확도는 코드북 품질과 LLM 능력에 의존
- 현재 Go 전용 (AST 분석, 어노테이션 포맷)
- 20B 로컬 모델이 한국어 프롬프트에서 "검증"→validate 키워드 편향을 보임
- 개별 LLM 스코어링(35회 호출)이 배치보다 느림 (~1분 vs ~5초)

## 8. 결론

AI 코드 에이전트는 무관한 컨텍스트를 처리하는 데 연산의 99%를 낭비한다. filefunc은 250토큰의 구조화된 메타데이터 — 8B 모델에 대한 코드북 쿼리 — 가 350,000토큰의 브루트포스 컨텍스트를 대체할 수 있음을 보여준다. 연산량 126,000배 감소.

핵심 통찰은 새로운 알고리즘이 아니라 컨벤션이다: 쓰기 시점에 통제된 어휘로 코드에 어노테이션을 달고, 검색이 아닌 분류를 하라. 이것이 소스 코드에 적용된 SILK 원칙이다: 구조가 검색을 제거한다.
